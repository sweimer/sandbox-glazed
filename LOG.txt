## PROJECT OVERVIEW

A Drupal 11 Lando site that embeds micro-frontend apps (React, Astro) as
Drupal blocks via a custom CLI toolkit at 3pd-ide/starter-scripts/cli/.

CLI command: `3pd`
Key commands:
  3pd astro app <name>          Create new Astro app
  3pd astro module --install    Build + install as Drupal block module
  3pd react app <name>          Create new React app
  3pd react module --install    Build + install as Drupal block module

## LANDO COMMANDS

lando crx   — Full cache reset. Does: rm files/php/ + TRUNCATE all cache tables
              + stop Apache + drush status (pre-warms container into DB) + restart Apache.
              Use after every lando rebuild AND after enabling/installing modules.

lando drush — Standard drush. DO NOT use for cache clearing. Use lando crx.


## CRITICAL: lando crx / cache reset procedure

After lando rebuild OR after enabling modules OR when drush hangs:
  1. lando crx        ← handles everything
  2. Open browser     ← site loads immediately

NEVER use `lando drush cr` — it hangs. See SANDBOX-glazed/.lando.yml for
the full crx command. Root cause: Apache workers deadlock compiling the Drupal
service container simultaneously when cache is empty. crx stops Apache first,
pre-warms via drush status, then restarts Apache.

If URLs don't resolve after lando restart:
  lando poweroff && lando start   (re-registers proxy routes)
  If still broken: lando rebuild -y, then immediately lando crx

## SESSION: 2026-03-27 — Strategy decisions (no code changes)

### 3PD repo strategy — decided
Each starter kit gets its own standalone git repo.
3PD devs clone only their framework's repo. No /web, no Drupal, no Lando required.
`3pd <framework> module` (without --install) packages the module subfolder only.
3PD dev pushes a feature branch. Drupal owner pulls + runs --install.
The --install flag is INTERNAL USE ONLY (copies to /web + enables in Drupal).

### Data persistence — decided
Apps that collect data (forms etc.) use Express + SQLite. NOT Drupal's DB.
Reasons:
  1. Portability: if client changes CMS, Express + SQLite file moves intact.
  2. Compliance: .gov/HUD site — data must stay within FedRAMP boundary.
     No external PaaS (Railway, Render etc.) for data storage on this project.
  3. Simplicity: SQLite is a file, zero infrastructure overhead vs PostgreSQL.

In production: Express backend co-deployed on the Drupal server, behind the
same reverse proxy / auth wall as Drupal. Module JS calls /api/app-name/ on
the same domain — already protected by the auth layer, no extra token needed.

SQLite file location:
  Dev:  ./server/db/app.sqlite  (relative, inside app folder)
  Prod: /var/app-data/<appname>/app.sqlite  (absolute, outside web root)

### Client context — HUD (.gov federal agency)
The Drupal site is behind a sign-in wall for a HUD federal agency.
Federal guidance applies (FedRAMP, FISMA, NIST 800-53, Section 508).
Apps ONLY exist as Drupal modules in production — never standalone.
All data infrastructure must stay within the compliance boundary.
No unauthorized cloud services. Data must be US-hosted on approved infra.

### Starter kit plan — agreed order
1. Finish 0.starter-astro (read-only SSG — nearly done)
2. Build 0.starter-astro-forms (Astro SSG + Express + SQLite)
3. Update 0.starter-react (apply all lessons: SQLite, version string, drush path, cors)

## 3PD WORKFLOW (third-party developer process)

3PD dev (no access to /web Drupal codebase):
  1. Clone their framework's starter repo (e.g. 0.starter-astro)
  2. Run: 3pd astro app <feature-name>
  3. Build the feature in their app folder
  4. Run: 3pd astro module   (no --install — packages module subfolder only)
  5. Push feature branch containing the module subfolder
  6. Open PR to main repo

Drupal owner (this repo):
  1. Pull the feature branch
  2. Run: 3pd astro module --install   (copies to /web/modules/custom + enables)
  3. Run: lando crx
  4. Place block in Layout Builder

The module subfolder is the deliverable. The app src/ is the development environment.

#### SESSION: 2026-03-28 (continued) — Lando bootstrap deadlock fix

##### Root cause
After `lando rebuild -y`, multiple modules with external HTTP calls (ai, ai_automators,
ai_api_explorer, ai_ckeditor, ai_observability, gemini_provider, announcements_feed,
update, dxpr_builder) all tried to make DNS lookups during bootstrap. Docker's DNS
resolver hung on the first lookup, deadlocking all Apache workers simultaneously.
The crx pre-warm procedure (stop Apache → drush status → start Apache) also hung
because drush itself was blocked on the same DNS call.

##### Fix applied
1. Added to settings.php (above Lando DB block):
     $settings['http_client_config']['timeout'] = 5;
   This bounds all Guzzle HTTP requests to 5 seconds, preventing DNS hangs from
   blocking the bootstrap indefinitely.

2. Added TRUNCATE TABLE router to crx command in .lando.yml — the router table
   was not being cleared by crx, causing "No route found" errors after cache clears.

3. Added router rebuild step to crx in .lando.yml:
     drush php:eval "\Drupal::service('router.builder')->rebuild();"
   NOTE: .lando.yml changes require `lando rebuild -y` to take effect.
   Current session used manual `drush php:eval` workaround after each crx.

##### lando rebuild -y — DONE (2026-03-28)
SSL fix (build_as_root nossl.cnf) is now applied. lando crx works again.

##### IMPORTANT: next lando rebuild required
.lando.yml now has router truncate + rebuild in crx, but this requires
another `lando rebuild -y` to take effect. Until then, after any lando crx
that clears the router, run manually:
  lando drush php:eval "\Drupal::service('router.builder')->rebuild();"

## SESSION: 2026-03-29 — Form persistence via Drupal DB (Option A)

### Problem
astro-forms apps deployed as Drupal modules had broken form submit + submissions
list (HTTP 404) because the module's static JS called /api/submissions on the
same origin (Drupal), but no route existed there. Express/SQLite only runs locally.

### Decision: Option A — Drupal handles the API
The Drupal module now exposes /api/submissions itself, backed by Drupal's MySQL DB.
This is the right fit for Pantheon (no separate process, same-origin, no CORS needed).
The local Express/SQLite remains for 3PD dev workflow (they don't run Drupal locally).
On --install, SQLite data is read and baked into the module as seed data.

### What was built

#### 3pd.config.json (new file, repo root)
Project-level config read by the CLI when creating new apps:
  {
    "pantheonUrl": "https://dev-3-pd-ide.pantheonsite.io",

  }

  1. Sets PUBLIC_DRUPAL_BASE_URL automatically (no manual .env edit)
  2. Replaces YOUR_CONTENT_TYPE in index.astro automatically (no manual edit)
New apps are fully functional out of the box.

#### create-module.js — major additions (0.starter-astro-forms)

1. SQLite reader (readSqlite)
   Writes a temp .mjs script into the app dir, runs it via node to read
   server/db/app.sqlite using better-sqlite3. Reads all rows + PRAGMA table_info
   from the submissions table. Returns empty gracefully if DB doesn't exist.

2. data/seed.json
   Written into every generated module. Contains SQLite columns + all rows.
   Committed to git with the module — bakes local dev data into the deploy artifact.

3. {machineName}.install
   - hook_schema(): defines the submissions table in Drupal's schema API.
     Column types derived from SQLite PRAGMA (INTEGER→serial/int, TEXT→text big,
     DATETIME/VARCHAR→varchar 255). Falls back to hardcoded defaults if no DB.
   - hook_install(): calls _import_seed() on first enable. Drupal creates the
     table from hook_schema() before hook_install() runs.
   - _import_seed(): truncates the table, inserts all rows from seed.json.
     Called by hook_install() on first install, and by post_deploy.php on every
     Pantheon deploy to keep seed data in sync.

4. src/Controller/SubmissionsController.php
   - list(): GET /api/submissions — SELECT * ORDER BY id DESC → JsonResponse
   - create(): POST /api/submissions — INSERT name+message+created_at → 201

5. .routing.yml — updated with API routes
   - {machineName}.api_submissions_list:  GET  /api/submissions
   - {machineName}.api_submissions_create: POST /api/submissions
   No CSRF token required (JsonResponse controllers, not form submissions).

#### private/scripts/post_deploy.php — updated
Scans /code/web/modules/custom/ for any module with a data/seed.json.
For each found, calls _import_seed() via drush php:eval.
Result: every Pantheon push re-syncs seed data for all hudx forms modules.

#### Critical fix: Adobe Muse verticalspacer infinite loop
Muse pages use <div class="verticalspacer" style="height: calc(-1359px + 100vh)">
to pin the footer to the viewport bottom. Inside an iframe, 100vh = iframe height.
This creates a circular loop: spacer expands → scrollHeight grows → iframe grows → repeat.
Result: iframe is always ~8770px regardless of page content.

Fix applied to hudx-resize.js:
  When inside an iframe, inject CSS: .verticalspacer { display: none !important; }
  This removes the spacer from layout before measuring scrollHeight.
  Footer flows naturally after main content. Height reflects real content per page.

  Drupal's #markup runs through Xss::filter() which removes <iframe> tags.
  Fix: use Markup::create() to mark the HTML as trusted.

### ✓ CloudFront caching workflow — understood and locked in

  Every S3 file upload requires a CloudFront invalidation to propagate.
  Without invalidation, CloudFront serves the previous cached version.

    2. CloudFront console → Distributions → E34CQM3GCYZN5R → Invalidations → Create → /*
    3. Wait ~30s → hard refresh (Cmd+Shift+R) to bypass browser cache

  AWS CLI credentials: scott.weimer IAM user has RequireMFAforUsers policy which blocks
  all CLI actions without MFA session token. Use S3/CloudFront console for all operations
  on this client account. Access keys exist but are insufficient without MFA.

### ✓ hudx-resize.js — final working state (all issues resolved)

  Root causes diagnosed via console.log debugging:
    1. Height locked at 5736px on every page (doc.scrollHeight)
       Muse sets an explicit height on the <html> element once, never updates across navigations.
       documentElement.scrollHeight is pinned to the tallest page loaded in the session.
       Fix: use body.scrollHeight ONLY — not Math.max(body, doc).
       body.scrollHeight correctly reflects each page's true content height.

    2. Height growing additively on each navigation (resize cascade)
       iframe height change → resize event fires in child → Muse recalculates → our handler
       fires immediately — order non-deterministic, can measure inflated height.
       Fix: debounce ALL events (resize, hashchange, popstate) through a single shared timer.
       600ms allows the postMessage → parent resize → child resize cascade to fully settle.

    3. Muse navigation: not SPA routing.
       Muse multi-page exports load separate .html files. hashchange/popstate do not fire.
       Navigation is detected via DOMContentLoaded on each new page load.
       resize events (from parent adjusting iframe height) are the main ongoing trigger.

    4. Accordion opens not detected
       Accordion opens change body height but don't trigger window events.
       Fix: ResizeObserver on document.body catches all DOM-level size changes,
       including accordion opens, without relying on window events.

  Final hudx-resize.js architecture:
    - collapseMuseSpacer(): setProperty 'important' on .verticalspacer (unchanged, works)
    - sendHeight(): body.scrollHeight only (was Math.max(body, doc))
    - debouncedSendHeight(): single shared 600ms debounce timer
    - Events: DOMContentLoaded → sendHeight() immediately (no debounce)
              resize / hashchange / popstate → debouncedSendHeight()
              ResizeObserver(body) → debouncedSendHeight()
    - window.hudxSendHeight exposed for manual calls

  Result: sc-training app loads correctly in iframe, resizes per page, accordions
  expand the iframe when opened. No additive extra space.

### #1 — Vanilla JS starter kit

  A lightweight starter for simple interactive apps that don't need a framework.
  No React, no Astro, no build step. Plain HTML + CSS + JS packaged as a Drupal block.
  Follows the same CLI pattern: 3pd vanilla app <name> / 3pd vanilla module --install
  Ideal for: simple widgets, calculators, lightweight tools, low-complexity embeds.
  Shares the same block generation, routing, and seeding patterns as other starters.


### #2 — Full-page app starter (--page mode)

  Current starters produce Block plugins that embed inside a Drupal page's chrome.
  A --page mode (or separate starter) would produce a Controller + bare page twig
  instead — the app owns the full page including header, footer, and nav.
  Drupal handles routing and auth only.

  Key differences from block mode:
    - No Plugin/Block/ class — Controller + routing.yml only
    - templates/page--hudx-[name].html.twig strips all Drupal regions
    - App bundles its own Bootstrap/USWDS + nav/header/footer components
    - Theme-independent by design (no drupal-dev-assets reliance in production)
    - Can be assigned as Drupal home page via /admin/config/system/site-information

  Implementation approach: --page flag on existing starters (same app code,
  different build config + different generator output) OR a separate page starter.
  Same CLI pattern: 3pd react module --page --install

  USWDS note: for .gov projects, full-page apps should use USWDS not Bootstrap
  (federal design standards, Section 508). This decision should be made before
  building the page starter.


### USWDS theme migration (Level 1 — HUDx)

  This is a HUD .gov site. USWDS is the federal standard (Section 508, FISMA,
  consistency with other HUD properties like hudexchange.info).
  Current starters depend on Bootstrap (hard) and DXPR theme CSS (soft).

  Migration scope:
    - drupal-dev-assets.js: swap Bootstrap/DXPR source for USWDS CSS/JS
    - Starter templates: replace Bootstrap class patterns with USWDS equivalents
      (btn btn-primary → usa-button, form-control → usa-input, row/col-* → usa-grid, etc.)
    - React apps: consider @trussworks/react-uswds component library (cleaner than
      class-name swapping, provides accessible USWDS components out of the box)
    - Existing apps (3pd-depot, checklist, ai-helper, embed-request): each needs
      component-level markup updates — mechanical work, not architectural
    - create-module.js files: no changes needed (don't generate UI markup)

  Not a rewrite — the framework patterns stay identical. Work is systematic
  find-and-replace of component patterns + dev assets source swap.


### 3PD as a reusable product (Level 2 — beyond HUDx)

  The 3PD framework is already generic. The HUDx implementation is one configuration.
  Gap to close for Level 2:

    Theme abstraction:
      - drupal-dev-assets becomes theme-configurable (Bootstrap, USWDS, or custom URL)
      - 3pd.config.json gains a themeSystem key (bootstrap | uswds | custom)
      - Starter templates ship with minimal/no styling — just structure and mount points
      - Theme-specific starter variants documented separately from framework docs

    Naming abstraction:
      - hudx_* prefix becomes configurable in 3pd.config.json (modulePrefix: 'hudx')
      - All generators read modulePrefix from config instead of hardcoding 'hudx'

    Documentation split:
      - Framework docs (starter patterns, CLI, generator architecture) → project-agnostic
      - HUDx implementation docs (PROMPT.txt files, LOG.txt) → HUDx-specific

    Deployment abstraction:
      - Pantheon-specific deploy pipeline generalized to support other hosts
      - 3pd.config.json gains deployTarget key

  The framework is already a product. This work is configuration abstraction,
  not a rewrite. Estimated complexity: medium. No core patterns change.

## SESSION: 2026-04-07 — Lando startup deadlock (documented for future reference)

### Problem: site unresponsive after lando poweroff + lando start

Root causes (two separate issues):

1. lando stop (not poweroff) leaves the shared Traefik proxy running with stale routing.
   Symptom: HTTP ECONNRESET or TIMEOUT on .lndo.site URLs after switching projects.
   Fix: ALWAYS use `lando poweroff` (not `lando stop`) when done with a Drupal project.
   `lando stop` only stops app containers. `lando poweroff` tears down the proxy too.

2. Apache bootstrap deadlock on cold start (empty cache + slow Drupal bootstrap).
   Symptom: site unresponsive (HTTP 000 / timeout) even though Apache workers are running.
   Root cause: `lando crx`'s drush warm-up step (`drush status`) hangs because multiple
   Drupal modules (ai, update, announcements_feed, dxpr_builder, etc.) make external HTTP
   calls during bootstrap. With $settings['http_client_config']['timeout'] = 30, each
   call can block up to 30s. Multiplied across 8+ modules = drush never finishes.
   If crx is run multiple times (or my repeated ssh attempts), stacked drush processes
   each try to compile the service container simultaneously → deadlock spiral.

### Manual recovery procedure (if crx hangs)

Step 1 — Kill ALL stacked drush processes:
  lando ssh -s appserver -c "pkill -9 -f drush"
  Verify: lando ssh -s appserver -c "ps aux | grep -c drush"  ← should be 1 (just grep)

Step 2 — Truncate caches directly via MySQL (skip drush):
  lando ssh -s appserver -c "mysql -h database -u drupal -pdrupal --skip-ssl drupal -e \
    'TRUNCATE TABLE cache_bootstrap; TRUNCATE TABLE cache_config; TRUNCATE TABLE cache_container; \
     TRUNCATE TABLE cache_data; TRUNCATE TABLE cache_default; TRUNCATE TABLE cache_discovery; \
     TRUNCATE TABLE semaphore;'"
  lando ssh -s appserver -c "rm -rf /app/web/sites/default/files/php/"

Step 3 — Temporarily drop Guzzle timeout to 3s in settings.php:
  Change: $settings['http_client_config']['timeout'] = 30;
  To:     $settings['http_client_config']['timeout'] = 3;
  NOTE: 30s is needed for Anthropic API calls (3pd-ai-coder). Restore after warmup.

Step 4 — Run a single clean drush status (Apache must be stopped first):
  lando ssh -s appserver -c "apache2ctl stop"
  lando ssh -s appserver -c "cd /app && bash vendor/bin/drush status --root=/app/web \
    --uri=http://sandbox-glazed.lndo.site"
  Wait for it to show the status table (Drupal version, DB Connected, etc.)
  If Apache was already stopped and stacks are clear, this takes ~10-15s with 3s timeout.

Step 5 — Force-restart Apache (kill all workers + respawn):
  lando ssh -s appserver -c "pkill -9 apache2; sleep 2"
  Apache master (PID in entrypoint) will respawn workers automatically.
  If not: lando ssh -s appserver -c "apache2ctl start"

Step 6 — Rebuild router:
  lando drush php:eval "Drupal::service('router.builder')->rebuild(); echo 'done';"

Step 7 — Restore Guzzle timeout in settings.php:
  Change: $settings['http_client_config']['timeout'] = 3;
  Back to: $settings['http_client_config']['timeout'] = 30;

Step 8 — Verify:
  curl -sk --max-time 15 https://sandbox-glazed.lndo.site/ -o /dev/null -w "HTTP %{http_code}\n"
  Should return HTTP 200 in 2-3s.

### Diagnosis commands (to understand the state before fixing)

  # Check if Apache is listening (look for :0050 LISTEN = port 80):
  lando ssh -s appserver -c "grep ':0050' /proc/net/tcp"

  # Count stacked drush processes:
  lando ssh -s appserver -c "ps aux | grep -c drush"

  # Test if Traefik can reach Apache (bypasses host DNS):
  docker exec landoproxyhyperion5000gandalfedition_proxy_1 \
    wget -qO- --timeout=5 http://172.19.0.3/ | head -3

  # Test if the site responds at all (bypasses DNS resolution):
  curl -sk --max-time 5 https://127.0.0.1/ -H "Host: sandbox-glazed.lndo.site" \
    -o /dev/null -w "HTTP %{http_code}\n"

### TODO: fix crx to be robust against drush warmup hangs

The crx command's drush status step should have a timeout so it never hangs indefinitely.
Change in .lando.yml crx cmd step 3:
  Current: cd /app && bash vendor/bin/drush status ...
  Better:  cd /app && timeout 20 bash vendor/bin/drush status ... 2>/dev/null || echo "warmup timed out, continuing"
This way crx always completes and Apache always restarts, even if drush hangs.
The first browser request will finish warming the cache (may be slow but won't deadlock).
Requires lando rebuild -y to take effect.

### Session/project workflow to prevent this

Two separate stacks — keep them mentally separate:

  Drupal/Lando stack (Docker containers):
    End session:  lando poweroff   ← NOT lando stop
    Start session: lando start, then lando crx
    poweroff tears down Traefik proxy + all containers.
    stop only pauses containers — proxy stays, routes go stale.

  Starter kit apps (React/Astro — Node processes on host Mac):
    End session:  3pd stop   (or Ctrl+C in the terminal running npm run dev)
    Start session: npm run dev (or 3pd run ai)
    These are host processes, completely independent of Lando/Docker.
    Port conflicts (4000-5175 range) come from forgetting to stop these,
    not from Lando.

  The two stacks don't conflict with each other. Drupal uses port 80/443 (via
  Traefik proxy). Starter kits use ports 4000-5175. They can all run simultaneously.
  The only conflict is between two Drupal/Lando projects sharing the same proxy —
  that's solved by lando poweroff between switches.
