## PROJECT OVERVIEW

A Drupal 11 Lando site that embeds micro-frontend apps (React, Astro) as
Drupal blocks via a custom CLI toolkit at 3pd-ide/starter-scripts/cli/.

CLI command: `3pd`
Key commands:
  3pd astro app <name>          Create new Astro app
  3pd astro module --install    Build + install as Drupal block module
  3pd react app <name>          Create new React app
  3pd react module --install    Build + install as Drupal block module


## ARCHITECTURE

Lando services (2 only — do NOT add more):
  appserver   → Drupal 11 at http://sandbox-glazed.lndo.site
  database    → MariaDB 10.6

IMPORTANT: The astro Lando service was REMOVED. It caused HSTS poisoning of
all *.lndo.site subdomains. Do NOT add it back. No framework gets a Lando service.

React apps: Express runs on the HOST machine via npm run dev. No Lando service.

## LANDO COMMANDS

lando crx   — Full cache reset. Does: rm files/php/ + TRUNCATE all cache tables
              + stop Apache + drush status (pre-warms container into DB) + restart Apache.
              Use after every lando rebuild AND after enabling/installing modules.

lando drush — Standard drush. DO NOT use for cache clearing. Use lando crx.


## CRITICAL: lando crx / cache reset procedure

After lando rebuild OR after enabling modules OR when drush hangs:
  1. lando crx        ← handles everything
  2. Open browser     ← site loads immediately

NEVER use `lando drush cr` — it hangs. See SANDBOX-glazed/.lando.yml for
the full crx command. Root cause: Apache workers deadlock compiling the Drupal
service container simultaneously when cache is empty. crx stops Apache first,
pre-warms via drush status, then restarts Apache.

If URLs don't resolve after lando restart:
  lando poweroff && lando start   (re-registers proxy routes)
  If still broken: lando rebuild -y, then immediately lando crx

## SESSION: 2026-03-27 — What we fixed / built

### astro---poc-02 — WORKING. Shows live Drupal data.

Root causes found and fixed:

1. HAProxy strips square brackets from query strings.
   `?fields[node--basic_page_layout_builder]=title` → `?fields=` (empty)
   Fix: removed the sparse fieldset param entirely. Plain URL with no params works.
   Note: %5B/%5D encoding also gets stripped by HAProxy — do NOT use brackets
   in query strings for Lando-proxied fetches.

2. Browser cache served stale JS after rebuild.
   `?v=1.x` was hardcoded in libraries.yml — never changed between builds.
   Fix: 0.starter-astro/create-module.js now uses Date.now() as version.
   Every rebuild produces a new ?v=<timestamp> cache buster.

3. Old starter template default URL was node/page — this site has no `page` nodes.
   Fix: poc-02 index.astro updated to fetch basic_page_layout_builder.
   The starter template (0.starter-astro) still uses node/page as placeholder —
   this is correct since it's generic. Apps must update it for their content type.

Block at node/6 now displays:

  - SANDBOX - Stracat
  - Sandbox - Layout Builder


## SESSION: 2026-03-27 — Strategy decisions (no code changes)

### 3PD repo strategy — decided
Each starter kit gets its own standalone git repo.
3PD devs clone only their framework's repo. No /web, no Drupal, no Lando required.
`3pd <framework> module` (without --install) packages the module subfolder only.
3PD dev pushes a feature branch. Drupal owner pulls + runs --install.
The --install flag is INTERNAL USE ONLY (copies to /web + enables in Drupal).

### Data persistence — decided
Apps that collect data (forms etc.) use Express + SQLite. NOT Drupal's DB.
Reasons:
  1. Portability: if client changes CMS, Express + SQLite file moves intact.
  2. Compliance: .gov/HUD site — data must stay within FedRAMP boundary.
     No external PaaS (Railway, Render etc.) for data storage on this project.
  3. Simplicity: SQLite is a file, zero infrastructure overhead vs PostgreSQL.

In production: Express backend co-deployed on the Drupal server, behind the
same reverse proxy / auth wall as Drupal. Module JS calls /api/app-name/ on
the same domain — already protected by the auth layer, no extra token needed.

SQLite file location:
  Dev:  ./server/db/app.sqlite  (relative, inside app folder)
  Prod: /var/app-data/<appname>/app.sqlite  (absolute, outside web root)

### Client context — HUD (.gov federal agency)
The Drupal site is behind a sign-in wall for a HUD federal agency.
Federal guidance applies (FedRAMP, FISMA, NIST 800-53, Section 508).
Apps ONLY exist as Drupal modules in production — never standalone.
All data infrastructure must stay within the compliance boundary.
No unauthorized cloud services. Data must be US-hosted on approved infra.

## 3PD WORKFLOW (third-party developer process)

3PD dev (no access to /web Drupal codebase):
  1. Clone their framework's starter repo (e.g. 0.starter-astro)
  2. Run: 3pd astro app <feature-name>
  3. Build the feature in their app folder
  4. Run: 3pd astro module   (no --install — packages module subfolder only)
  5. Push feature branch containing the module subfolder
  6. Open PR to main repo

Drupal owner (this repo):
  1. Pull the feature branch
  2. Run: 3pd astro module --install   (copies to /web/modules/custom + enables)
  3. Run: lando crx
  4. Place block in Layout Builder

The module subfolder is the deliverable. The app src/ is the development environment.
