# 3PD IDE — Astro Forms Starter Kit: AI Guide
# =========================================================
# Read this file before building on or modifying this starter,
# or before creating a new forms-capable starter for another
# framework. Documents architecture decisions and lessons learned.
# =========================================================


## WHAT THIS STARTER IS

An extension of the read-only Astro starter that adds a data persistence
layer for apps that need to collect or store data (forms, submissions, etc.).

Frontend:  Astro SSG (same as 0.starter-astro)
Dev backend:   Express + SQLite (better-sqlite3) — local only
Prod backend:  Drupal REST controller — generated by --install, backed by Drupal's MySQL

The Drupal module contains the static build (HTML/CSS/JS) AND a PHP REST
controller that handles /api/submissions. No separate server needed in production.


## CONTENT UPDATE MODEL

This starter has three distinct layers, each with a different update path:

  DRUPAL CONTENT (nodes, entities fetched via JSON:API)
  → Updated by editors in the CMS. Reflected immediately on next page load.
  → No redeployment needed.

  FORM SUBMISSIONS (rows in the submissions table)
  → Written by end users at runtime. Stored in Drupal's MySQL DB (prod)
     or SQLite (local dev). No redeployment needed.

  APP STRUCTURE (form fields, labels, layout, logic in src/)
  → Lives in source code. Changes require a new `3pd astro-forms module` build
     and redeployment of the module.

Use this starter when: the app needs both CMS-driven content AND the ability
to collect/store user data, and editors need control over the content side.


## DUAL PERSISTENCE MODEL

### Local dev (3PD developer)
- Express + SQLite runs on the host machine via `npm run dev`
- Dev submits test data, confirms schema works
- SQLite file lives at server/db/app.sqlite (gitignored)

### Deployed (Drupal module on Pantheon/prod)
- `--install` reads SQLite, bakes rows into data/seed.json (committed)
- `hook_schema()` creates the submissions table in Drupal's MySQL on first enable
- `hook_install()` seeds the table from seed.json on first enable
- `SubmissionsController.php` handles GET + POST /api/submissions same-origin
- post_deploy.php re-seeds on every Pantheon push (for dev/staging only)

### Data flow direction
Pantheon dev is the DB of record once the module is live.
Before starting new local dev work: pull current data from Pantheon.
  3pd astro-forms db pull
  (fetches GET /api/{APP_SLUG}/{table} from Pantheon, writes rows to server/db/app.sqlite)
  After pull: restart dev server — lsof -ti:3001 | xargs kill -9 && npm run dev
Do NOT blindly push seed.json from a stale local SQLite — it will truncate
Pantheon's current data on the next deploy.


## WHY TWO BACKENDS

Express + SQLite for local dev:
1. 3PD devs have no Drupal locally — they need a lightweight backend to test forms
2. Portability: if the client changes CMS, the data layer is independent
3. Compliance: for .gov/HUD deployments, all data stays on the same server

Drupal DB for deployed module:
1. No separate process to manage on Pantheon (no Express, no PM2, no proxy config)
2. Same-origin — no CORS needed in production
3. drush updb handles schema migrations automatically on deploy

See LOG.txt "Session 2026-03-29" for full decision context.

## 3pd.config.json (repo root)

Project-level config read by the CLI when creating new apps:
  {
    "pantheonUrl": "https://dev-3-pd-ide.pantheonsite.io",
    "defaultContentType": "basic_page_layout_builder"
  }

Both `3pd astro-forms app` and `3pd astro app` read this automatically:
  1. Sets PUBLIC_DRUPAL_BASE_URL to pantheonUrl (no manual .env edit)
  2. Replaces YOUR_CONTENT_TYPE in index.astro with defaultContentType

New apps are fully functional out of the box. Update 3pd.config.json when
the Pantheon URL or default content type changes.


## CLI INTEGRATION

Commands:
  3pd astro-forms app <name>          → commands/astro-forms-app.js
  3pd astro-forms module              → commands/astro-forms-module.js  (block mode)
  3pd astro-forms module --install    → internal mode: copies to /web + enables
  3pd astro-forms page                → commands/astro-forms-page.js   (full-page mode)
  3pd astro-forms page --install      → internal mode: copies to /web + enables

Framework prefix (shared/frameworks.js):
  FRAMEWORK_PREFIXES.astroForms = 'astro-forms---'
  Example output folder: astro-forms---my-app


## HOW `3pd astro-forms app <name>` WORKS

1. Reads 3pd.config.json from repo root
2. Slugifies name → prefixes with 'astro-forms---'
3. Copies starter-template/ skipping node_modules
4. Creates .env from .env.example, sets PUBLIC_DRUPAL_BASE_URL from pantheonUrl
5. Replaces YOUR_CONTENT_TYPE in index.astro with defaultContentType
6. Patches package.json "name" field
7. Runs npm install

NOTE: better-sqlite3 compiles a native binary. Requires Python + Xcode CLI tools.
If npm install fails with a gyp error, check Node version compatibility.


## HOW `3pd astro-forms page --install` WORKS (FULL-PAGE MODE)

Same Astro app code as block mode. Only the generator output differs.

1. Reads SQLite from server/db/app.sqlite (same as block mode)
2. Builds Astro app (npm run build)
3. Generates dist/page.html:
   - Reads dist/index.html from Astro build
   - Rewrites /_astro/ asset paths → /modules/custom/{machineName}/dist/assets/
   - Injects Bootstrap 5.3.x CSS + Bootstrap Icons CDN <link> into <head>
   - Injects Bootstrap JS CDN <script> before </body>
   - Reason: the Drupal theme CSS is never loaded when bypassing the theming layer
4. Generates module in app dir: {machineName}/
   - data/seed.json, .install, SubmissionsController.php, routing.yml (same as block)
   - dist/assets/ — renamed stable JS/CSS from Astro build
   - dist/page.html — rewritten full HTML (see above)
   - PageController.php — returns new Response(file_get_contents('dist/page.html'))
   - .module — minimal (no hook_theme, no hook_theme_suggestions_page_alter)
   - NO templates/ directory
5. Copies module to /web/modules/custom/{machineName}/
6. drush pm:uninstall → drush en → lando crx → router rebuild
7. Reports page URL: /<appName>

WHY RAW RESPONSE (not hook_theme_suggestions_page_alter):
  Drupal only searches for page--*.html.twig templates in the ACTIVE THEME's
  templates/ folder — never in modules. A module can add a page suggestion but
  cannot provide the template file. The raw Symfony Response approach bypasses
  this entirely: PageController reads dist/page.html and returns it directly,
  skipping Drupal's theming layer, BigPipe, and the admin toolbar.


## HOW `3pd astro-forms module --install` WORKS

1. Reads SQLite from server/db/app.sqlite via a temp .mjs script
2. Builds Astro app (npm run build)
3. Generates module in app dir: {machineName}/
   - data/seed.json (SQLite rows)
   - .install (hook_schema + hook_install + _import_seed)
   - SubmissionsController.php (GET list + POST submit)
   - .routing.yml with test page + API routes
   - Block plugin, HudxTestController, twig template, libraries.yml
4. Copies module to /web/modules/custom/{machineName}/
5. drush pm:uninstall (ensures hook_schema + hook_install re-run)
6. drush en (creates table from hook_schema, seeds from hook_install)
7. lando crx (cache clear)
8. lando drush php:eval router rebuild
9. Reports test URL: /hudx-test/<appname>


## ROUTE NAMESPACING — APP_SLUG

All API routes are namespaced per app using APP_SLUG:
  /api/{APP_SLUG}/submissions
  e.g. /api/astro-forms---my-app/submissions

APP_SLUG is set automatically when `3pd astro-forms app <name>` runs.
  - .env: APP_SLUG=astro-forms---my-app (used by Express)
  - .env: PUBLIC_APP_SLUG=astro-forms---my-app (used by Astro/Vite build)

The route collision between multiple forms apps on the same Drupal is resolved.
Multiple astro-forms modules can coexist.

The table name is also detected dynamically from sqlite_master (not hardcoded).
readSqlite() returns { rows, cols, tableName } — used throughout create-module.js.
Routes in routing.yml: /api/${appName}/${tableName}


## MULTI-PAGE SUPPORT (page mode only)

`create-page.js` automatically discovers sub-pages from the Astro build output.
Any `dist/{slug}/index.html` (excluding `api-reference`, `styleguide`, `_astro`, `assets`)
becomes a Drupal route at `/{appName}/{slug}` with its own PageController method.

Steps to add a sub-page:
  1. Create `src/pages/{slug}.astro` in the app
  2. Run `3pd astro-forms page [--install]` — no other config needed
  3. The generator creates `dist/{slug}.html`, a `subpage_{slug}()` method in PageController,
     and a `{machineName}.subpage_{slug}` route in routing.yml automatically

No limit on number of sub-pages. `api-reference` and `styleguide` are always excluded.


## DYNAMIC NAV (page mode — MenuController)

`create-page.js` always generates `MenuController.php`. It reads a named Drupal menu
via the `menu.link_tree` service (depth 2, enabled links only) and returns JSON:

  GET /api/{appName}/menu/{menu_name}
  → [{ title, url, weight, children: [{ title, url, weight }] }]

Local dev equivalent is in `server/server.js`:
  GET /api/${APP_SLUG}/menu/:menuName → reads `server/menu.json`

`server/menu.json` ships with a minimal `main` menu. Update it to mirror your
Drupal menu structure for accurate local dev. Its structure is:
  { "main": [{ "title": "", "url": "", "weight": 0, "children": [] }] }

Nav JS should: fetch on page load, fall back to static HTML if the request fails.
Use `Cache-Control: no-store` (already set in MenuController) to keep nav fresh.


## AUTH-GATED EDIT BUTTONS (page mode)

`PageController.php` injects `data-drupal-auth="1"` on `<body>` for authenticated users.
`Layout.astro` simulates this in local dev via `{isDev && <script>}`.

Pattern for auth-gated edit buttons in app CSS:
  .edit-btn { display: none; }
  body[data-drupal-auth] .edit-btn { display: inline-flex; }

This single CSS rule controls all edit buttons site-wide — no JS auth checks needed.
Edit buttons open Drupal admin URLs in an iframe modal (see AlexRenew demo for reference).


## DRUPAL DEV ASSETS (Bootstrap + DXPR theme in local dev)

During local dev the app loads Bootstrap and DXPR theme CSS/JS from two generated files:
  public/drupal-dev-styles.css   ← Bootstrap + DXPR theme CSS concatenated
  public/drupal-dev-scripts.js   ← Bootstrap bundle JS

These files are GITIGNORED and dev-only. They are conditionally included in Layout.astro
using `import.meta.env.DEV` which Astro statically replaces with `false` during
`astro build` — so these references are compiled away and never appear in the Drupal module.

The files are generated automatically when a new app is created via `3pd astro-forms app <name>`.
If the Drupal theme changes, run from inside the app directory:
  3pd styles refresh   ← regenerates public/drupal-dev-styles.css + public/drupal-dev-scripts.js
  npm run dev          ← restart to pick up changes

Or to regenerate for ALL apps at once (from the 3pd-ide root):
  3pd styles sync

Bootstrap Icons are loaded from CDN (jsdelivr.net) — not from the local file,
because relative font-file paths in the npm package would break in the public/ context.

DXPR theme JS is excluded on purpose: its scripts require Drupal.behaviors and
drupalSettings globals that don't exist outside of a Drupal page context.
Only bootstrap.bundle.min.js is included (handles dropdowns, modals, tooltips, etc.).

## EXPRESS SERVER DESIGN

server.js binds to 127.0.0.1 (not 0.0.0.0):
  - In dev: still reachable from the same machine (localhost = 127.0.0.1)
  - In prod: not needed — Drupal's SubmissionsController handles /api/submissions
  - Never expose the Express port directly to the internet

CORS is configured for the Astro dev server origin (localhost:4321).

Routes (local dev only):
  GET  /api/{APP_SLUG}/submissions   → SELECT * ORDER BY created_at DESC → JSON array
  POST /api/{APP_SLUG}/submissions   → INSERT, return saved row as JSON (201)


## SQLITE DESIGN

better-sqlite3 is synchronous — no async/await needed for DB calls.
WAL journal mode: allows concurrent reads while a write is in progress.
Schema is applied via CREATE TABLE IF NOT EXISTS on every startup.

DB_PATH env var:
  Dev:  ./server/db/app.sqlite  (relative to app root via process.cwd())

The .sqlite file (and WAL journal .sqlite-shm, .sqlite-wal) are gitignored.
Never commit them. The seed.json in the module IS committed — it is the
snapshot of local data at --install time.


## SUBMISSIONS CONTROLLER (generated PHP)

Location: web/modules/custom/{machineName}/src/Controller/SubmissionsController.php

  list()   → GET  /api/{appName}/submissions → SELECT * FROM {tableName} ORDER BY id DESC
  submit() → POST /api/{appName}/submissions → INSERT name + message + created_at

Table name: {machineName}_{tableName} where tableName is auto-detected from sqlite_master.
  e.g. hudx_astro_forms___app_12_submissions

IMPORTANT: Do NOT name the POST method `create()` — ControllerBase has a static
create() factory method. Use `submit()` to avoid the PHP fatal error.


## LOCAL create-module.js OVERRIDE

astro-forms-module.js checks the current app directory for a create-module.js
before falling back to the starter kit's generic one.

Use this for apps that need a custom controller, schema, or route structure
without forking the starter kit. Place create-module.js in the app root.
Example: astro-forms---3pd-checklist has its own create-module.js that generates
ChecklistController.php with 3 routes instead of SubmissionsController.php.

The twig fix (always use body HTML from dist/index.html) is applied in BOTH
the starter create-module.js AND should be replicated in any custom override.


## TWIG GENERATION (critical for Astro SSG)

Astro apps with interactive JS produce an external bundle (hoisted.*.js in _astro/).
The twig template MUST contain the full body HTML from dist/index.html — not a bare
mount div. Without it, the JS runs but finds none of its target elements (white page).

Rule: always extract body HTML from dist/index.html. Strip <script> and <link> tags.
Libraries.yml handles asset loading — inline tags are not needed in twig.

  const twig = bodyHtml ? `${bodyHtml}\n` : `<div id="${mountId}"></div>\n`;

Symptoms of wrong twig: white page on Pantheon, JS in browser network tab loads fine.


## ENV VARS REFERENCE

PUBLIC_DRUPAL_BASE_URL  Drupal site URL for JSON:API reads. Auto-set from 3pd.config.json.
                        Empty in production (.env.production).
PUBLIC_API_BASE_URL     Express server URL for dev. Empty in production (same-origin).
APP_SLUG                App folder name. Used by Express for route namespacing.
                        Set automatically by `3pd astro-forms app <name>`.
PUBLIC_APP_SLUG         Same as APP_SLUG — Vite-exposed for Astro fetch URLs.
PORT                    Express port. Default: 3001.
DB_PATH                 SQLite file path. Default: ./server/db/app.sqlite.
CORS_ORIGIN             Astro dev server URL. Default: http://localhost:4321.

PUBLIC_* vars are baked into the Astro/Vite build at compile time.
Server vars (PORT, DB_PATH, etc.) are read at Express startup time.


## PANTHEON DEPLOY FLOW

On push to main → GH Actions → Pantheon:
1. composer install → vendor/ pushed to Pantheon
2. Quicksilver (post_deploy.php): drush updb → router rebuild → drush cr
   → scans modules/custom/*/data/seed.json
   → calls _import_seed() for each found (truncates + reimports)
   NOTE: seed import on non-first deploys only works if module is already enabled.
   First deploy: GH Actions terminus pm:enable → hook_install() seeds the table.
3. GH Actions terminus: pm:list --disabled | grep hudx_ → pm:enable new modules

Test URL on Pantheon: https://dev-3-pd-ide.pantheonsite.io/hudx-test/<appname>


## KNOWN ISSUES / LESSONS LEARNED

1. better-sqlite3 requires ^11.0.0+ for Node 23. v9.x fails to compile.

2. HAProxy strips square brackets from query strings.
   NEVER use bracket notation in fetch URLs. See 0.starter-astro/PROMPT.txt.

3. version in libraries.yml uses Date.now(). Do not hardcode version: 1.x.

4. ControllerBase::create() is a static factory method. The POST handler in
   SubmissionsController must be named submit(), not create().

5. The router must be explicitly rebuilt after module enable. lando crx alone
   is not sufficient until lando rebuild -y is run to pick up .lando.yml changes.
   create-module.js now runs `lando drush php:eval router.builder->rebuild()`
   from the repo root (requires cwd: path.dirname(drupalWebRoot)).

6. /api/submissions route conflict: RESOLVED (2026-03-29). Routes are now namespaced
   per app using APP_SLUG: /api/{APP_SLUG}/{table}. Multiple forms apps can coexist.

7. astro.config.mjs @hudx / @hudxjs aliases use monorepo-relative paths.
   Will break in standalone repo. Fix when implementing standalone repos.

8. api-reference.astro: raw {} in <pre> code blocks are parsed as JSX expressions
   by Astro, even inside <pre>. All literal { } in code examples must be escaped
   as &#123; / &#125;. The esbuild error message points to the wrong line (source
   map inaccuracy) — search the whole file for unescaped { } in HTML context.

9. page--*.html.twig in modules is silently ignored by Drupal. Drupal only finds
   page-level twig overrides in the active theme's templates/ folder. Use a raw
   Symfony Response from the controller instead (see create-page.js).

## HOW TO BUILD A NEW FORMS STARTER FOR A DIFFERENT FRAMEWORK

Use this starter as your reference. Key differences to account for:

  - Does the framework's build output work with the same create-module.js?
    → React: no (uses Vite, different bundle structure — see 0.starter-react)
    → Vue/Svelte: likely yes, same Vite external bundle pattern as Astro

  - Does the framework need a router that conflicts with Drupal?
    → React: BrowserRouter → MemoryRouter required
    → Vue Router: use hash mode

  - The Express server pattern is IDENTICAL across all frameworks.
    Copy server/ verbatim. Only the frontend changes.

  - The .install + SubmissionsController pattern is IDENTICAL.
    Copy from create-module.js. Resolve the /api/submissions route conflict
    BEFORE implementing (see known issue above).

  - Apply all items in the mandatory checklist above.
