## WHAT THIS STARTER IS

A Drupal micro-frontend pattern using React + Vite on the frontend,
Express + SQLite (better-sqlite3) on the backend. The starter produces a
Drupal block module that embeds the React SPA into any Layout Builder region.

Frontend:  React + Vite
Backend:   Express + SQLite (better-sqlite3)
Data:      App-owned SQLite file — NOT Drupal's database

Same data persistence rationale as 0.starter-astro-forms. See that PROMPT.txt
for the full reasoning (portability, FedRAMP compliance).

## CLI INTEGRATION

Commands:
  3pd react app <name>          → commands/react-app.js
  3pd react module              → commands/react-module.js
  3pd react module --install    → internal mode (copies to /web + enables)

Framework prefix (shared/frameworks.js):
  FRAMEWORK_PREFIXES.react = 'react---'
  Example output folder: react---my-app


## HOW `3pd react app <name>` WORKS

1. Slugifies name → prefixes with 'react---'
2. Copies starter-template/ skipping node_modules
3. Creates .env from .env.example (root level, not server/)
4. Patches package.json "name" field
5. Runs npm install
   NOTE: better-sqlite3 compiles a native binary. Requires Python + Xcode CLI tools.
   If npm install fails with a gyp error, check Node version (needs Node 18+, ^11.0.0).

No database setup step. SQLite file is created automatically on first server startup.


## HOW `3pd react module` WORKS (create-module.js)

Naming conventions (derived from app folder name):
  appName:      react---my-app
  machineName:  hudx_react___my_app       (hudx_ prefix, hyphens→underscores)
  hyphenName:   hudx-react---my-app       (underscores→hyphens)
  mountId:      hudx-react---my-app-root  (used as the DOM element id)
  className:    HudxReact3pdMyAppBlock    (PascalCase for PHP class)

Steps:
1. REWRITES src/main.jsx with a Drupal-safe mount pattern:
   - Mounts React into the element with id="${mountId}"
   - Guards against double-mounting (el.__hudxReactRoot)
   - Waits for DOMContentLoaded
   - Registers Drupal.behaviors.${behaviorName} for Layout Builder AJAX

2. ENFORCES MemoryRouter in src/App.jsx via AST transform (@babel/parser):
   - Removes BrowserRouter imports and wraps
   - Replaces with MemoryRouter (BrowserRouter breaks inside Drupal blocks)

3. REWRITES index.html: changes <div id="root"> → <div id="${mountId}">

4. Runs `npm run build` (Vite always produces external bundles in dist/assets/)
   Vite reads .env.production automatically — VITE_API_BASE_URL is blank.

5. Renames hashed bundles to stable filenames:
   dist/assets/index-[hash].js  → dist/assets/hudx_react___my_app.js
   dist/assets/index-[hash].css → dist/assets/hudx_react___my_app.css

6. Creates module directory at: apps/react---my-app/hudx_react___my_app/

7. Generates Drupal module files:
   - hudx_react___my_app.info.yml
   - hudx_react___my_app.module      (implements hook_theme)
   - hudx_react___my_app.libraries.yml  (version: Date.now(), external bundles)
   - hudx_react___my_app.routing.yml    (test route: /hudx-test/<appName>)
   - src/Plugin/Block/...Block.php
   - src/Controller/HudxTestController.php
   - templates/hudx-react---my-app-block.html.twig  → <div id="${mountId}"></div>

8. In --install mode:
   - Copies module to web/modules/custom/hudx_react___my_app/
   - Enables via: lando drush en / lando drush php:eval fallback
   - Clears caches: lando crx
   - Prints test page URL


## DEV WORKFLOW

  cd apps/react---my-app
  npm run dev
  → Starts both Express (127.0.0.1:4000) and Vite dev server (localhost:5173)
  → API:  http://127.0.0.1:4000
  → UI:   http://localhost:5173
  → SQLite file created at server/db/app.sqlite on first request

Individual servers (debugging):
  npm run dev:server   → Express only
  npm run dev:client   → Vite only


## EXPRESS SERVER DESIGN

server.js binds to 127.0.0.1 (not 0.0.0.0):
  - In dev: reachable from the same machine (localhost = 127.0.0.1)
  - In prod: not publicly accessible — only via reverse proxy
  - Never expose the Express port directly to the internet

CORS is configured for the Vite dev server origin (localhost:5173).

Routes:
  POST /api/test/add   → INSERT test_entries, return saved row (201)
  GET  /api/test/all   → SELECT * ORDER BY created_at DESC → JSON array


## SQLITE DESIGN

Same pattern as 0.starter-astro-forms. See that PROMPT.txt for full details.
Key differences for React:
  - DB_PATH env var read via dotenv.config() (no path arg — reads root .env)
  - Schema: INTEGER PRIMARY KEY AUTOINCREMENT (not SERIAL — SQLite syntax)
  - created_at: TEXT DEFAULT (datetime('now')) (not TIMESTAMP DEFAULT NOW())


## ENV VAR PATTERN

React/Vite uses VITE_ prefix (not PUBLIC_) for client-exposed vars.

.env (dev):
  VITE_API_BASE_URL=http://localhost:4000   ← baked into Vite dev build
  PORT=4000
  HOST=127.0.0.1
  DB_PATH=./server/db/app.sqlite
  CORS_ORIGIN=http://localhost:5173

.env.production (production — Vite reads automatically during `vite build`):
  VITE_API_BASE_URL=                        ← blank, same-origin in Drupal

In Home.jsx:
  const API_BASE = import.meta.env.VITE_API_BASE_URL || '';


## PRODUCTION DEPLOYMENT

Same as 0.starter-astro-forms. See that PROMPT.txt.
Key difference: Vite dev server default port is 5173 (not 4321).


## KNOWN ISSUES / LESSONS LEARNED

1. BrowserRouter breaks in Drupal. The module generator enforces MemoryRouter via AST.
   Never use BrowserRouter in the starter template — the generator strips it anyway.

2. better-sqlite3 requires ^11.0.0 for Node 23. v9.x fails to compile.

3. Vite always produces external bundles for React (dist/assets/*.js + *.css).
   No need to handle the inline script case (Astro-specific).

4. version in libraries.yml uses Date.now() — never a static string.

5. CORS on localhost:5173 (Vite default). If the 3PD dev changes the Vite port,
   they must update CORS_ORIGIN in .env.

6. The Express server is NOT in the Drupal module. Deploy it separately via PM2.
   See 0.starter-astro-forms/PROMPT.txt for the full production deployment notes.

7. No square brackets in any fetch URLs — HAProxy strips them.
