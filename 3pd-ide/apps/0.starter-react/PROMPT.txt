# Read this file before building on or modifying this starter,

## WHAT THIS STARTER IS

A Drupal micro-frontend pattern using React + Vite on the frontend,
Express + SQLite (better-sqlite3) on the backend. The starter produces a
Drupal block module that embeds the React SPA into any Layout Builder region.

Frontend:  React + Vite
Backend:   Express + SQLite (better-sqlite3)
Data:      App-owned SQLite file — NOT Drupal's database

Same data persistence rationale as 0.starter-astro-forms. See that PROMPT.txt
for the full reasoning (portability, FedRAMP compliance).


## CONTENT UPDATE MODEL

This starter does NOT fetch content from Drupal at runtime. All content
(copy, labels, layout, UI logic) lives in the React source code.

  APP CONTENT (text, labels, UI structure in src/)
  → Lives in source code. Any content change — even copy edits — requires
     a new `3pd react module` build and redeployment of the module.
  → Editors cannot update content without a developer and a deployment.

  FORM SUBMISSIONS (rows in the submissions table)
  → Written by end users at runtime. Stored in Drupal's MySQL DB (prod)
     or SQLite (local dev). No redeployment needed.

Use this starter when: the app requires complex client-side state, routing,
or interactivity that Astro cannot provide, AND content changes are infrequent
or are managed by developers rather than editors.

If content needs to be editable by non-developers, consider 0.starter-astro-forms
instead — it fetches CMS content at runtime while still supporting form submissions.

## CLI INTEGRATION

Commands:
  3pd react app <name>          → commands/react-app.js
  3pd react module              → commands/react-module.js
  3pd react module --install    → internal mode (copies to /web + enables)
  3pd react db pull             → commands/react-db-pull.js (pulls Pantheon data → local SQLite)

Framework prefix (shared/frameworks.js):
  FRAMEWORK_PREFIXES.react = 'react---'
  Example output folder: react---my-app


## HOW `3pd react app <name>` WORKS

1. Slugifies name → prefixes with 'react---'
2. Copies starter-template/ skipping node_modules
3. Creates .env from .env.example (root level, not server/)
   - Injects APP_SLUG and VITE_APP_SLUG from the folder name
   - Reads 3pd.config.json (pantheonUrl) if present — can be used to pre-populate Drupal URL
4. Patches package.json "name" field
5. Runs npm install
   NOTE: better-sqlite3 compiles a native binary. Requires Python + Xcode CLI tools.
   If npm install fails with a gyp error, check Node version (needs Node 18+, ^11.0.0).

No database setup step. SQLite file is created automatically on first server startup.


## HOW `3pd react module` WORKS (create-module.js)

Naming conventions (derived from app folder name):
  appName:         react---my-app
  machineName:     hudx_react___my_app         (hudx_ prefix, hyphens→underscores)
  hyphenName:      hudx-react---my-app          (underscores→hyphens)
  mountId:         hudx-react---my-app-root     (used as the DOM element id)
  className:       HudxReact3pdMyAppBlock       (PascalCase for PHP class)
  drupalTableName: hudx_react___my_app_submissions  (machineName + _ + tableName)

Steps:
1. REWRITES src/main.jsx with a Drupal-safe mount pattern:
   - Mounts React into the element with id="${mountId}"
   - Guards against double-mounting (el.__hudxReactRoot)
   - Waits for DOMContentLoaded
   - Registers Drupal.behaviors.${behaviorName} for Layout Builder AJAX

2. ENFORCES MemoryRouter in src/App.jsx via AST transform (@babel/parser):
   - Removes BrowserRouter imports and wraps
   - Replaces with MemoryRouter (BrowserRouter breaks inside Drupal blocks)

3. REWRITES index.html: changes <div id="root"> → <div id="${mountId}">

4. Runs `npm run build` (Vite always produces external bundles in dist/assets/)
   Vite reads .env.production automatically — VITE_API_BASE_URL is blank.

5. Renames hashed bundles to stable filenames:
   dist/assets/index-[hash].js  → dist/assets/hudx_react___my_app.js
   dist/assets/index-[hash].css → dist/assets/hudx_react___my_app.css

6. Reads SQLite via temp .mjs script (readSqlite):
   - Reads submissions table columns + rows via PRAGMA table_info + SELECT *
   - Returns { rows, cols, tableName } — empty gracefully if DB doesn't exist

7. Generates data/seed.json — {table, columns, rows} from SQLite.
   Baked into the module for Drupal to import on install.

8. Creates module directory at: apps/react---my-app/hudx_react___my_app/

9. Generates Drupal module files:
   - hudx_react___my_app.info.yml
   - hudx_react___my_app.module       (implements hook_theme)
   - hudx_react___my_app.libraries.yml   (version: Date.now(), external bundles)
   - hudx_react___my_app.routing.yml     (test route + GET/POST /api/{appName}/{table})
   - hudx_react___my_app.install         (hook_schema, hook_install, _import_seed)
   - src/Plugin/Block/...Block.php
   - src/Controller/HudxTestController.php
   - src/Controller/SubmissionsController.php  (list + submit, mirrors Express routes)
   - templates/hudx-react---my-app-block.html.twig  → <div id="${mountId}"></div>

API routes in routing.yml (namespaced by appName to prevent conflicts):
   GET  /api/{appName}/{table}   → SubmissionsController::list()
   POST /api/{appName}/{table}   → SubmissionsController::submit()

10. In --install mode:
    - Copies module to web/modules/custom/hudx_react___my_app/
    - Enables via: lando drush en / lando drush php:eval fallback
    - Clears caches: lando crx
    - Rebuilds router: lando drush php:eval "\Drupal::service('router.builder')->rebuild();"
    - Prints test page URL


## DEV WORKFLOW

  cd apps/react---my-app
  npm run dev
  → Starts both Express (127.0.0.1:4000) and Vite dev server (localhost:5173)
  → API:  http://127.0.0.1:4000
  → UI:   http://localhost:5173
  → SQLite file created at server/db/app.sqlite on first request

Individual servers (debugging):
  npm run dev:server   → Express only
  npm run dev:client   → Vite only


## EXPRESS SERVER DESIGN

server.js binds to 127.0.0.1 (not 0.0.0.0):
  - In dev: reachable from the same machine (localhost = 127.0.0.1)
  - In prod: not publicly accessible — only via reverse proxy
  - Never expose the Express port directly to the internet

CORS is configured for the Vite dev server origin (localhost:5173).

Routes use APP_SLUG from .env for namespacing (prevents conflicts when multiple apps installed):
  GET  /api/{APP_SLUG}/submissions  → SELECT * FROM submissions ORDER BY id DESC → JSON array
  POST /api/{APP_SLUG}/submissions  → INSERT message, return saved row (201)

In the generated Drupal module, SubmissionsController.php mirrors these routes.
The JS fetch URL uses the same /api/{APP_SLUG}/submissions path — works in both dev and prod.


## SQLITE DESIGN

Same pattern as 0.starter-astro-forms. See that PROMPT.txt for full details.
Key differences for React:
  - DB_PATH env var read via dotenv.config() (no path arg — reads root .env)
  - Schema: INTEGER PRIMARY KEY AUTOINCREMENT (not SERIAL — SQLite syntax)
  - created_at: TEXT DEFAULT (datetime('now')) (not TIMESTAMP DEFAULT NOW())
  - Table: submissions with columns: id, message, created_at


## ENV VAR PATTERN

React/Vite uses VITE_ prefix (not PUBLIC_) for client-exposed vars.

.env (dev):
  APP_SLUG=react---my-app             ← injected by react-app.js on app creation
  VITE_APP_SLUG=react---my-app        ← same, VITE_ prefix so Vite exposes to browser
  VITE_API_BASE_URL=http://localhost:4000   ← baked into Vite dev build
  PORT=4000
  HOST=127.0.0.1
  DB_PATH=./server/db/app.sqlite
  CORS_ORIGIN=http://localhost:5173

.env.production (production — Vite reads automatically during `vite build`):
  VITE_API_BASE_URL=                        ← blank, same-origin in Drupal

In Home.jsx:
  const API_BASE = import.meta.env.VITE_API_BASE_URL || '';
  const APP_SLUG = import.meta.env.VITE_APP_SLUG     || '';
  // fetch calls use: ${API_BASE}/api/${APP_SLUG}/submissions

## LOCAL create-module.js OVERRIDE

react-module.js checks the current app directory for a create-module.js before
falling back to the starter kit's generic one:

  const localCreateModulePath = path.join(cwd, 'create-module.js');
  const starterCreateModulePath = path.join(ideRoot, 'apps', '0.starter-react', 'create-module.js');
  const createModulePath = fs.existsSync(localCreateModulePath) ? localCreateModulePath : starterCreateModulePath;

Use this for apps that need custom PHP controllers, extra routes, or different schema
without forking the starter kit. Example: react---3pd-ai-helper has its own
create-module.js that generates GenerateController.php + HistoryController.php.


## KNOWN ISSUES / LESSONS LEARNED

1. BrowserRouter breaks in Drupal. The module generator enforces MemoryRouter via AST.
   Never use BrowserRouter in the starter template — the generator strips it anyway.

2. better-sqlite3 requires ^11.0.0 for Node 23. v9.x fails to compile.

3. Vite always produces external bundles for React (dist/assets/*.js + *.css).
   No need to handle the inline script case (Astro-specific). Twig template is always
   a bare <div id="${mountId}"></div> — no body HTML extraction needed.

4. version in libraries.yml uses Date.now() — never a static string.

5. CORS on localhost:5173 (Vite default). If the 3PD dev changes the Vite port,
   they must update CORS_ORIGIN in .env.

6. The Express server is NOT in the Drupal module. Deploy it separately via PM2.
   See 0.starter-astro-forms/PROMPT.txt for the full production deployment notes.

7. No square brackets in any fetch URLs — HAProxy strips them.

8. APP_SLUG namespacing: all API routes use /api/{APP_SLUG}/submissions to prevent
   conflicts when multiple React apps are installed in the same Drupal.
   APP_SLUG is injected by react-app.js into .env on app creation.
   Never hardcode the route path — always read from VITE_APP_SLUG.

9. Drupal table naming: drupalTableName = ${machineName}_${tableName}
   e.g. hudx_react___my_app_submissions
   Avoids conflicts across modules in Drupal's shared database.

10. `3pd react db pull` pulls live data from /api/{APP_SLUG}/{table} on Pantheon.
    Uses VITE_DRUPAL_BASE_URL or VITE_API_BASE_URL from .env, or pantheonUrl from 3pd.config.json.
    After pull, must restart dev server: lsof -ti:4000 | xargs kill -9 && npm run dev

11. `3pd react db pull` SSL bypass for Lando URLs.
    Node.js rejects self-signed Lando certs (*.lndo.site). react-db-pull.js sets
    NODE_TLS_REJECT_UNAUTHORIZED=0 when the URL contains 'lndo.site' or 'localhost',
    then restores to '1' after the fetch. Do not leave this globally disabled.

12. PHP template literal escaping in custom create-module.js.
    JS only interpolates `${...}`. Bare `$variable` in a template literal is safe and
    passes through as-is to the PHP file. Use `$variable` (not `\\$variable`) for PHP
    variables in template strings. Keep `\\Drupal`, `\\Exception`, `\\PDO` (namespace
    separators) as `\\` since those really do need a literal backslash in the PHP output.

13. Pantheon API key delivery pattern.
    Store secrets in Pantheon Secrets dashboard (Runtime + Web scope).
    In settings.php:
      if (isset($_ENV['PANTHEON_ENVIRONMENT']) && function_exists('pantheon_get_secret')) {
        $key = pantheon_get_secret('MY_SECRET');
        if (!empty($key)) { putenv('MY_SECRET=' . $key); }
      }
    For local dev: use settings.local.php (gitignored) with putenv('MY_SECRET=value').
    Uncomment the settings.local.php include in settings.php.
    HTTP timeout: increase $settings['http_client_config']['timeout'] to 30 or more
    for apps that call external APIs (default 5s is too short for AI endpoints).

11. seed.json / hook_install() / SubmissionsController.php follow the same pattern as
    0.starter-astro-forms. See that PROMPT.txt for full implementation details.
